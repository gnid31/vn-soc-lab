## 1. Vì sao detect cái này

`lsass.exe` (Local Security Authority Subsystem Service) lưu plain-text passwords, NTLM hashes, Kerberos tickets trong memory. Mọi credential-dumping technique đều phải mở handle vào lsass.exe với quyền `PROCESS_VM_READ`:

- **Mimikatz** (`sekurlsa::logonpasswords`) — toolkit nổi tiếng nhất.
- **ProcDump** với flag `-ma lsass.exe lsass.dmp` (LOLBin nhưng dùng cho dump).
- **Cobalt Strike** `mimikatz` module.
- **Atomic Red Team T1003.001**.

Sysmon event_id 10 ghi nhận mọi `OpenProcess` call. Khi target là `lsass.exe` và `GrantedAccess` chứa các bit nguy hiểm, đó là chỉ dấu mạnh.

**Lý do severity Critical:** dump LSASS thành công = attacker có toàn bộ credential active session trên máy → lateral movement / domain admin trong vài phút.

## 2. KQL query

```
event.code: "10" AND
winlog.provider_name: "Microsoft-Windows-Sysmon" AND
winlog.event_data.TargetImage.keyword: *\\lsass.exe AND
winlog.event_data.GrantedAccess: (
  "0x1010" OR
  "0x1410" OR
  "0x1438" OR
  "0x143a" OR
  "0x1fffff" OR
  "0x1f1fff"
)
```

**Giải thích:**

- `event.code: "10"` — Sysmon process access.
- `TargetImage.keyword: *\\lsass.exe` — `.keyword` bắt buộc do path chứa `\` ([bài học R1 §2a](R1-T1059.001-powershell-encoded.md#2a-bài-học-quan-trọng--khi-nào-phải-dùng-keyword)).
- `GrantedAccess`: các bit access right phổ biến của credential dumping:

  | Hex | Decoded | Tool điển hình |
  |---|---|---|
  | `0x1010` | `PROCESS_QUERY_LIMITED_INFORMATION + PROCESS_VM_READ` | Mimikatz default |
  | `0x1410` | + `PROCESS_VM_OPERATION` | Mimikatz with patching |
  | `0x1438` | + `PROCESS_DUP_HANDLE` | ProcDump style |
  | `0x143a` | + `PROCESS_VM_WRITE` | Injection scenarios |
  | `0x1fffff` | `PROCESS_ALL_ACCESS` | ProcDump `-ma` |
  | `0x1f1fff` | Variant ALL_ACCESS | Cobalt Strike beacon |

**Tại sao không bắt mọi GrantedAccess:** ngày bình thường có hàng nghìn legit OpenProcess vào lsass.exe (Defender, lsass tự loop, perfmon, Task Manager) chỉ với `PROCESS_QUERY_LIMITED_INFORMATION` (`0x1000`). Bắt cả `0x1000` → flood false-positive.

## 3. Ví dụ event match (sẽ trigger ở Pha 4 — Atomic T1003.001)

```yaml
event.code: "10"
winlog.provider_name: "Microsoft-Windows-Sysmon"

winlog.event_data.CallTrace    : "C:\\Windows\\SYSTEM32\\ntdll.dll+a5294|..."
```

## 4. False-positive thường gặp

| FP scenario | Đặc trưng | Khuyến nghị lọc |
|---|---|---|
| **Microsoft Defender (`MsMpEng.exe`) real-time scan** — ✅ observed Pha 4 (1 alert) | SourceImage `*\\Windows Defender\\MsMpEng.exe`, GrantedAccess `0x1010` (PROCESS_QUERY_LIMITED_INFORMATION + PROCESS_VM_READ) | Tune Pha 5+: `NOT SourceImage.keyword: *\\Windows Defender\\MsMpEng.exe`. **Lưu ý security:** attacker có thể inject Defender — phải verify Microsoft signature, không chỉ path |
| EDR như SentinelOne / CrowdStrike | Source = agent process đã ký hãng | Whitelist sau khi xác định hash đã ký |
| `wmiprvse.exe` query lsass for security info | Có thể `0x1410` | Hiếm — chấp nhận cho lab |
| Task Manager mở Detail tab | Thường `0x1000` only | Đã loại |

**Khuyến nghị:** giữ rule rộng để bắt Mimikatz mọi variant. Pha 4 sẽ tính FP rate thực tế → tinh chỉnh.

## 6. Smoke-test trong Pha 3

⚠️ **R2 không có smoke-test benign trong Pha 3.**

Lý do: muốn trigger Sysmon e10 với GrantedAccess `0x1410` cần thực sự `OpenProcess` vào lsass.exe với quyền nguy hiểm. Mọi binary làm điều đó:

- Bị Windows Defender chặn ngay (Mimikatz, ProcDump targeting lsass).
- Cần Administrator + tắt Defender — không phù hợp Pha 3 (Defender phải bật).

→ **R2 sẽ smoke-test ở Pha 4** khi setup Atomic Red Team chính thức + snapshot Win10 + tạm tắt Defender đúng quy trình:

